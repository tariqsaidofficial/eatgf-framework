## 1. Purpose

This document establishes the unified visual, typographic, and user experience standards for all EATGF documentation rendered through the Docusaurus governance portal and consumed as raw Markdown in the authority repository. Compliance with this guide ensures consistent readability, professional governance presentation, and scan-ability across all 8 framework layers.

## 2. Scope

This guide applies to:

- All Markdown documents within the `eatgf-framework` authority repository
- The Docusaurus portal configuration (`governance-docs-site/portal/`)
- Sidebar structure, navigation, and footer content
- Custom CSS, component styling, and homepage content
- Admonition usage, table formatting, and emoji discipline

This guide does not govern external content, third-party integrations, or ISO standard documents referenced by the framework.

## 3. Definitions

| Term | Definition |
|------|-----------|
| Admonition | A visually distinct callout block in Docusaurus (rendered via `:::type` syntax) used to highlight governance rules, warnings, references, and implementation notes. |
| Sidebar | The left-hand navigation panel in the Docusaurus portal that provides document hierarchy and navigation. |
| Scan-ability | The quality of a document that allows a reader to quickly identify key sections, decisions, and action items without reading the full text. |
| Visual Hierarchy | The arrangement of headings, spacing, and emphasis that communicates the relative importance and structure of content. |
| Infima | The CSS framework used by Docusaurus for base styling. |

## 5. Heading Hierarchy

### 5.1 Rules

| Level | Usage | Formatting |
|-------|-------|-----------|
| H1 (`#`) | Document title only — exactly one per document | Descriptive title without filename, version suffix, or emoji |
| H2 (`##`) | Major sections — numbered sequentially | `## 1. Purpose`, `## 2. Scope` — no emoji prefixes |
| H3 (`###`) | Subsections within an H2 | `### 5.1 Rules` — numbered relative to parent H2 |
| H4 (`####`) | Tertiary detail within an H3 | Use sparingly — indicates content may benefit from restructuring |
| H5/H6 | Prohibited | If H5/H6 is needed, restructure into separate documents or subsections |

### 5.2 Prohibited Heading Patterns

| Pattern | Violation | Correction |
|---------|-----------|-----------|
| `#  Enterprise Governance Framework` | Emoji in H1 | `# Enterprise Governance Framework` |
| `##  What's Included` | Emoji in H2 | `## 1. Contents` |
| `## 1 Foundation Layer` | Number emoji in H2 | `## 1. Foundation Layer` |
| `# OFFICIAL_DESIGNATION.md` | Filename as H1 | `# Official Designation` |
| `# BASELINE_DECLARATION_v1.0.md` | Version and extension in H1 | `# Baseline Declaration` |

## 6. Sidebar Structure

### 6.1 Current State Assessment

The portal sidebar (`portal/sidebars.ts`) uses `autogenerated` from the directory structure, producing raw folder names (e.g., `00_FOUNDATION`, `01_MANAGEMENT_SYSTEMS`). Documents lack `sidebar_position` and `sidebar_label` frontmatter, resulting in unclear ordering and display names.

The root sidebar (`sidebars.ts`) references 30+ documents that do not exist, representing aspirational content.

### 6.2 Required Sidebar Configuration

Each framework layer directory shall contain a `_category_.json` file:

```json
{
  "label": "Foundation",
  "position": 1,
  "collapsible": true,
  "collapsed": false,
  "link": {
    "type": "doc",
    "id": "README"
  }
}
```

**Recommended sidebar labels:**

| Directory | Sidebar Label | Position |
|-----------|--------------|----------|
| `00_FOUNDATION` | Foundation | 1 |
| `01_MANAGEMENT_SYSTEMS` | Management Systems | 2 |
| `02_CONTROL_ARCHITECTURE` | Control Architecture | 3 |
| `03_GOVERNANCE_MODELS` | Governance Models | 4 |
| `04_POLICY_LAYER` | Policies | 5 |
| `05_DOMAIN_FRAMEWORKS` | Domain Frameworks | 6 |
| `06_AUDIT_AND_ASSURANCE` | Audit & Assurance | 7 |
| `07_REFERENCE_AND_EVOLUTION` | Reference & Evolution | 8 |

### 6.3 Document Frontmatter Standard

Every framework document shall include YAML frontmatter:

```yaml
---
sidebar_position: 1
sidebar_label: "Master Control Matrix"
title: "Master Control Matrix"
description: "Single source of authority for all 35 EATGF controls"
---
```

### 6.4 Depth vs Scan-ability Rules

- **Maximum sidebar depth:** 3 levels (Layer → Subdirectory → Document)
- **Maximum items per level:** 8 — if a directory exceeds 8 items, group by subcategory
- **Historical artifacts:** Collapse by default (`collapsed: true`) to reduce visual noise

## 7. Admonition Specification

### 7.1 Standard EATGF Admonition Types

No Docusaurus admonitions (`:::`) are currently used across the framework. The following 5 types are mandated for all governance documentation:

#### Governance Rule

```markdown
:::caution[Governance Rule]
This control mandates [requirement]. Non-compliance triggers escalation per EATGF-EDM-GOV-01.
:::
```

**Rendered appearance:** Orange/amber left border with  icon. Use for mandatory compliance statements, control requirements, and governance mandates.

#### Enforcement

```markdown
:::danger[Enforcement]
Deviation from this policy requires written approval from [authority]. Unauthorized modifications shall be reported to the Governance Steering Committee.
:::
```

**Rendered appearance:** Red left border with  icon. Use for non-compliance consequences, escalation triggers, and enforcement actions.

#### Reference

```markdown
:::info[Reference]
This requirement aligns with ISO/IEC 27001:2022 Clause 6.1.2 and maps to COBIT 2019 APO12.
:::
```

**Rendered appearance:** Blue left border with  icon. Use for cross-references to external standards (ISO, COBIT, NIST), related EATGF documents, and MCM control traceability.

#### Implementation Note

```markdown
:::tip[Implementation Note]
For Startup Edition deployments, this control may be implemented using a simplified checklist format. Refer to GOVERNANCE_BY_TEAM_SIZE.md for edition-specific guidance.
:::
```

**Rendered appearance:** Green left border with  icon. Use for practical implementation guidance, edition-specific adaptations, and deployment recommendations.

#### Audit Perspective

```markdown
:::note[Audit Perspective]
Evidence for this control shall include timestamped configuration exports and approval records. Refer to EVIDENCE_REGISTER_MASTER.md for column specifications.
:::
```

**Rendered appearance:** Gray left border with  icon. Use for auditor-facing evidence requirements, assessment criteria, and audit trail specifications.

### 7.2 Admonition Usage Rules

1. Maximum 3 admonitions per document section (H2 level) to prevent callout fatigue.
2. Admonitions shall not contain tables — use inline text or bulleted lists only.
3. Admonitions shall not nest (no admonition within an admonition).
4. Every policy document shall contain at least one `:::caution[Governance Rule]` admonition.
5. Every control document shall contain at least one `:::info[Reference]` admonition mapping to external standards.

### 7.3 Portal Configuration Requirement

The portal Docusaurus configuration currently uses `format: 'md'`. To enable admonition rendering, verify that the `remark-admonitions` plugin is active. If admonitions do not render correctly, update `portal/docusaurus.config.ts`:

```typescript
docs: {
  // Change from 'md' to 'detect' to enable admonition parsing
  remarkPlugins: [],
}
```

Standard Docusaurus admonitions (`:::note`, `:::tip`, `:::info`, `:::caution`, `:::danger`) are supported in plain Markdown mode without MDX. Confirm rendering in a test build before proceeding with framework-wide adoption.

## 8. Table Formatting Standards

### 8.1 Column Alignment

| Data Type | Alignment | Example |
|-----------|-----------|---------|
| Text (descriptions, names) | Left-align (`:---`) | `| Description |` |
| Status indicators | Center-align (`:---:`) | `|  |` |
| Numeric values | Right-align (`---:`) | `| 35 |` |
| Dates | Left-align | `| 2026-02-14 |` |
| Control IDs | Left-align | `| EATGF-DSS-SEC-01 |` |

### 8.2 Status Indicators

| Indicator | Meaning | Usage |
|-----------|---------|-------|
|  | Implemented / Compliant / Complete | Status columns, compliance matrices |
|  | Partial / In Progress / Requires Attention | Status columns, risk indicators |
|  | Not Implemented / Non-Compliant / Missing | Status columns, gap analyses |
| N/A | Not Applicable | Applicability matrices |

### 8.3 Table Rules

1. All tables shall include a header row.
2. Tables exceeding 8 columns shall be split or restructured into multiple tables.
3. Empty cells shall contain `—` (em dash), not be left blank.
4. Control ID columns shall always use the `EATGF-[DOMAIN]-[CATEGORY]-[NUMBER]` format.
5. Version columns shall follow semantic versioning format (`vX.Y.Z`).

## 9. Emoji Discipline

### 9.1 Permitted Emoji Usage

| Context | Permitted Emoji | Notes |
|---------|----------------|-------|
| Status tables |    | Status indicators only |
| Admonition titles |      | Per Section 7.1 definitions |
| Archived documents | Any | Historical artifacts in Layer 07 are exempt |

### 9.2 Prohibited Emoji Usage

| Context | Prohibition |
|---------|------------|
| H1 headings | No emoji under any circumstances |
| H2/H3 headings | No emoji — use numbered sections |
| Body text of policies | No decorative emoji |
| Body text of controls | No decorative emoji |
| Commit messages | No emoji |

### 9.3 Examples of Violations

| Current | Correction |
|---------|-----------|
| `##  Layer Purpose` | `## 1. Layer Purpose` |
| `##  Key Objectives` | `## 2. Key Objectives` |
| `##  Security` | `## 3. Security` |
| `Done! ` | `Complete.` |
| `## 1 Foundation` | `## 1. Foundation` |

## 10. Section Separator Discipline

### 10.1 Horizontal Rule Usage (`---`)

| Position | Required | Example |
|----------|----------|---------|
| Before Document Control block | Yes | `---\n**Document Control**` |
| Before Authority Sign-Off block | Yes | Implicit (follows Document Control) |
| Between the metadata table and Section 1 | Yes | After Framework Header Signature |
| Between H2 sections | No — use whitespace only | Standard Markdown paragraph spacing |
| After H1 title | No | Prohibited — heading provides its own visual separation |

### 10.2 Spacing Rules

- One blank line before every heading (H2, H3, H4).
- One blank line after every heading.
- One blank line before and after every table.
- One blank line before and after every admonition block.
- No double blank lines anywhere in the document.
- No trailing whitespace on any line.

## 11. Version Display

### 11.1 Current State

No Docusaurus versioned docs are implemented. The planned versioning strategy (documented in SUBMODULE_OPERATIONS.md) has not been executed. No version badge, dropdown, or banner exists in the portal UI.

### 11.2 Required Implementation

1. Execute `npx docusaurus docs:version 1.0` in the portal to snapshot the current framework baseline.
2. Add a version dropdown to the navbar configuration in `portal/docusaurus.config.ts`:

```typescript
navbar: {
  items: [
    {
      type: 'docsVersionDropdown',
      position: 'right',
    },
  ],
},
```

3. Display framework version in the footer: `EATGF v1.0 — Foundation Edition`
4. Each document's Framework Header Signature table includes the document version. The portal version represents the framework-wide edition.

### 11.3 Version Badge

Add a version badge to the portal homepage hero section:

```
EATGF v1.0 | Foundation Edition | Baseline Frozen 2026-02-14
```

| Feature | Title | Description |
|---------|-------|-------------|
| 1 | 35 Governance Controls | Single control authority via the Master Control Matrix, mapped to ISO 27001, ISO 42001, and COBIT 2019 |

| 3 | 3 Scalable Editions | Startup, SaaS, and Enterprise editions with role-based governance adapted to organizational maturity |

### 12.2 Portal URL

Update `portal/docusaurus.config.ts` from `your-docusaurus-site.example.com` to the production URL.

### 12.3 Footer Links

Replace placeholder community links (Stack Overflow, Discord, X/Twitter pointing to Docusaurus defaults) with:

- **Documentation:** Link to the Framework landing page
- **Source:** Link to the `eatgf-framework` GitHub repository

Remove the "Community" footer column if no community channels exist.

## 13. Governance Enforcement Rules

1. All documents merged into the authority repository shall conform to the visual and structural standards defined in this guide.
2. Pull requests that introduce new documents without YAML frontmatter (Section 6.3) shall not be approved.
3. Admonitions shall be used in accordance with Section 7 — no alternative callout mechanisms (emoji-prefixed headers, bold-prefix bullets) are permitted in new documents.
4. Emoji usage outside the permitted contexts defined in Section 9.1 constitutes a style violation and shall be corrected prior to merge.
5. The portal homepage shall not display default Docusaurus boilerplate content in any production deployment.
